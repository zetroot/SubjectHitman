- Спецификация на разработку компонента поиска СКИ по 5791-у
	- # Функциональные требования
		- ## Основополагающий документы
			- 218-ФЗ о кредитных историях https://normativ.kontur.ru/document/1/506681-federalnyy-zakon-ot-30-12-2004-n-218-fz
			- указание Банка России 5791-у https://normativ.kontur.ru/document?moduleId=1&documentId=490474
			- положение Банка России 758-п о формировании кредитных историй https://normativ.kontur.ru/document/1/506160-polozhenie-tsb-rf-ot-11-05-2021-n-758-p
		- ## Основное назначение системы
			- учитывать использование бесплатных кредитных отчёты
			- предоставлять функциональность запроса количества использованных бесплатных кредитных отчётов.
			- Заказ кредитного отчета в процессинге, его обработка и предоставление клиенту не входят в скоуп работ по разработке этого компонента.
			- ### Сценарии использования
				- #### Запрос количества использованных бесплатных отчётов.
					- Внешняя система делает запрос, содержащий сведения входящие в запрос кредитного отчёта.
					- Разрабатываемая система должна идентифицировть субъекта в своей базе данных с использованием алгоритма разработанного на основе 5791-у
					- Если субъекта нет - его необходимо создать. Если субъект найден - необходимо обновить/дополнить сведения о нем в БД системы.
					- Далее система должна подсчитать сколько бесплатных отчётов было использовано у этого субъекта в течении текущего календарного года.
						- При этом в случае, если у субъекта есть несколько бесплатных отчётов в течении настраиваемого промежутка времени (cooldown period) - система  должна учесть их как один отчёт. Эта функциональность нужна для защиты от дублирующихся заказов, когда клиент фактически не получил новой информации.
				- #### Учёт использованных отчётов
					- Каждый запрос отчёта идентифицируется неким идентификатором формата uuid - reportId
					- Обработка отчёта во всей системе выполняется по паттерну сага.
					- Сага обработки отчёта начинается с поступления заказа от внешней системы.
					- Сага завершается, когда выполнены все шаги и оркестратор оповещает о готовности отчёта
					- Сага может завершиться неуспешно на любом из шагов, такие отчёты нельзя учитывать как выданные субъекту.
					- Таким образом, функциональность учёта использованных отчётов должна начинать собственную сагу при получении события "отчёт заказан"
						- Это событие содержит всю информацию о запросе кредитного отчёта и метаинформацию - является ли он бесплатным.
					- При начале саги система должна идентифицировать субъекта по данным из запроса с использованием алгоритма разработанного на основе 5791-у и сохранить внутренний идентификатор субъекта в состоянии саги.
					- Сага учёта отчётов завершается учётом отчёта как "списан" если общая сага обработки отчёта завершается успешно и вызывающая система оповещается о готовности отчёта с указанным reportId.
					- Сага учёта отчётов завершается учётом отчёта как "не списан". если общая сага обработки отчёта завершается неуспешно и вызываются система оповещается о том что отчёт не может быть изготовлен.
					- Предусмотреть обработку timeout саги учёта отчётов - если сага долго не завершается необходимо проверить статус отчёта в основной системе. Если удалось получить определённый статус (успех или провал) - завершить сагу и учесть отчёт в соответствующем статусе (списан или не списан). Если статус отчёта неустановлен - увеличить счётчик проверок и запланировать новое timeout сообщение. Если счётчик проверок превышает определённое значение (max timeout retries - брать из настроек приложения) - завершить сагу и учесть отчёт как не списанный.
		- ## алгоритм идентификации субъектов.
			- Сократить количество ключей поиска
				- В Запросе КО нет информации про ОГРНИП
				- Нет различий между текущим и предыдущим ФИО, никаких дополнительных квалифицирующих признаков вводить не надо. То есть субъект просто имеет набор ФИО, неважно когда они действовали.
				- То же самое для документов удостоверяющих личность.
				- Исходя из этого необходимо объединить ключи поиска в которых матчинг происходит по ФИО+ДУЛ и предыдущие ФИО + предыдущий ДУЛ
			- ### Описание алгоритма
				- Структура данных субъекта
					- Субъект (мастер запись). Содержит uuid идентификатор и возможно какую то дополнительную метаинформацию
						- ФИО - фамилия, имя, отчество (пустые строки заменять на "-") - допустимы множественные вхождения
						- ДУЛ - документ удостоверяющий личность. код вида документа, серия, номер, дата выдачи. Допустимы множественные вхождения
						- Дата рождения - 0..1
						- ИНН - 0..1
						- СНИЛС - 0..1
						- ключи поиска - fk на субъекта, тип ключа поиска, sha256 от сконкатенированной строки полученной из персональных данных субъекта
				- При добавлении/обновлении субъекта в БД - система пересчитывает все ключи поиска. Если какого-то показателя нет - ключи поиска с этим показателем не расчитываются. Учесть что "-" для ИНН и СНИЛС означает отсутствие такового показателя.
				- Поиск субъекта в БД осуществляется по всем ключам сразу. В случае если для запроса подходит несколько субъектов - необходимо выбрать того, у которого наибольшее количество совпадений. Если количество совпадений одинаковое, то надо выбрать того, у которого совпадения по более сильным ключам. Если после этого субъект не идентифицирован однозначино - выбрать более старую запись.
	- ## Нефункциональные требования
		- Используемый стек технологий:
			- asp.net core web api (minimal api)
			- postgresql
			- ef core
			- wolverine (для подписки на сообщения, для обработки http вызовов). В качестве транспорта можно использовать rabbitmq или кафку
